1. Projektgrundlagen

Versionsnummer: 3

1.1 Einführung

Bei dem Projekt handelt es sich um einen inviduellen interaktiven Studienplaner (Kurz SPAsS), für Studenten der Hochschule Rhein Main. In diesem Tool soll es möglich sein, für den eigenen Studienverlauf einen Plan zu generieren, der bestandene Module, Fortschrittsregelung und fachliche Abhängigkeiten berücksichtigt.

© We acknowledge that this document uses material from the arc42 architecture template, https://arc42.org.

1.2. Ziele

  • Eine grafische Benutzungsoberfläche, die Drag & Drop bearbeitung ermöglicht.
    • Einheitliches Stylesheet / Design
  • In dieser Benutzungsoberfläche sollen enthalten sein:
    • Übersicht über den Studienfortschritt (Nicht-/bestandene Module farblich unterlegt) sowie höhere Semester direkt sichtbar.
    • Mögliche Abhängigkeiten der Module sollen per Knopfdruck einsehbar sein.
    • Zu jedem Modul sollen außerdem Informationen gespeichert sein, wie beispielsweise der Dozent, Raum, Fehlversuche und die Note.
    • Der aktuelle Notenschnitt soll ausgegeben werden.
    • Eine Wunschnotenschnitt soll eingetragen werden können, dann werden zu den aktuell belegten Modulen Noten berechnet, welche erreicht werden müssen, um auf diesen Wunschschnitt zu kommen.
  • Ausgabe eines übersichtlichen und lesbaren Studienplan als PDF

1.3. Anwendungsszenarios

  • Benutzer kann einen eigenen Semesterplan für seinen Studiengang anlegen.
  • Möglichkeit vorgebene Module zu verändern, hinzuzufügen oder zu Löschen.
  • Aus den Studienabhängigen Modulen soll es möglich sein, sich einen Semesterplan zu erstellen.
  • Das Eintragen von Noten soll möglich sein.
  • Eine Notenschnitt Prognose für das aktuelle Semester soll (wenn gewünscht) möglich sein.
  • Es soll möglich sein, Modulabhängigkeiten direkt einzusehen.
  • Berstandene Module kann der User markieren.

3.5. Anwendungsfall E

Modulabhängigkeiten anzeigen

Der Benutzer kann sich die Modulabhängigkeiten auf zwei Ebenen anzeigen lassen.

Modulansichtsauswahl Semesterplan

Die Modulansichtsauswahl sammelt in einem Container alle existierenden Module für den ausgewählten Studiengang.

Der Semesterplan zeigt die Module an, die sich der User ausgewählt hat.
User

Fachlicher Auslöser

  1. Der User wählt sich für das kommende Semester neue Module aus und möchte sehen, ob seine bereits erlangte Erfahrung in bestandenen Modulen ihm helfen werden.
  2. Der User überlegt sich im nächsten Semester ein Modul nicht zu belegen und möchte sehen, ob die verlorene Erfahrung für andere Module zu einem Problem kommen könnte.

Vorbedingungen

  • User hat einen Studiengang in seine Modulauswahl geladen.
Ablauf

Da Modulansicht und Semesterplan zur selben Zeit geöffnet sein können, geschieht die jeweilige Aktion im ausgewählten Container. (AND OR)

Wenn in der Modulauswahl Abhängigkeiten für das Modul "Mathematik" hervorgehoben werden, werden diese nicht im Semesterplan hervorgehoben. Es kann aber zeitgleich auch im Semesterplan Abhängigkeiten von einem anderen Fach (oder demselben) angezeigt werden - oder andersherum.
  1. Benutzer:
    • Öffnet (AND OR):
      1. Modulansicht
      2. Semesterplan
  2. System:
    • Listet die Module auf.
  3. Benutzer:
    • Macht einen Rechtsklick auf ein Modul
  4. System:
    • UI zeigt Kontextmenü an.
  5. Benutzer:
    • Wählt im Kontextmenü Abhängigkeiten anzeigen aus
  6. System:
      • UI blendet Module, die keine Abhängigkeiten haben, aus.
      • UI hebt Module, die Abhängigkeiten besitzen, hervor.
  7. Benutzer:
    • Macht einen Rechtsklick in die/den: (AND OR)
      1. Modulansicht
      2. Semesterplan
  8. System:
    • UI zeigt Kontextmenü an.
  9. Benutzer:
    • Wählt im Kontextmenü Abhängigkeiten nicht mehr anzeigen aus.
  10. System:
    • UI setzt Semesterplan (AND OR) Modulansicht in den Normalzustand zurück.

5. Bausteinsicht

5.1. Whitebox Gesamtsystem

Erläuterung

Der Studienplaner SPAsS ist nach den Regeln des MVC aufgebaut. Wir besitzen unsere MainView, welche aufgerufene Sub-Szenen initalisiert und managed. MainView, sowie die unteren Sub-Szenen greifen auf die Controller zu, welche wiederum auf unsere Daten zugreifen.

Die MainView ist die äußere Komponente unserer GUI. In dieser wird ein neuer Studienplan erstellt oder ein bestehender eingeladen. Diese kann auf alle Sub-Szenen zugreifen, beispielsweise wird die Sub-Szene ModulListView dann geöffnet, wenn der User sich in der MainView per Mausclick alle Module zu seinem Semesterplan anzeigen lassen möchte. Die MainView hält also die initialisation eines Studienplans, sowie das Kontextmenü der Application und ruft je nach User eingabe die Sub-Szenen auf. Ausnahme ist hier die PlanView, welche permanent nach aufrufen eines Studienplanes angezeigt wird.

Wie das aussehen könnte, kann man sich im Abschnitt 4.2. Benutzungsschnittstelle anschauen.

Die PlanView enthält den eigentlichen Stundenplan, es greift auf die Klasse StundenplanVerwaltung zu, welche den Stundenplan beinhaltet (Ausgewählte Module des Users), enthalten sowie eine semZahl (Wie viele Semester werden im Plan angezeigt) und das stundenplanLayout (Anordnungsfilter der Module). Die Klasse implementiert außerdem das Stundenplan Interface, welches im PlanController neue Module hinzufügen und entfernen kann.

Die SubSzene ModulListView ist dafür da, dass alle Module des Studiengangs angezeigt werden können. Die View greift auf die ModulListe zu, welche alle existierenden Module beinhält. Diese Klasse, kann die einzelnen Module filtern. Einzelnen Module beinhalten:

Lehrveranstaltung

Ist eine Lehrveranstaltung(Objekt) Liste, da ein Modul mehrere Lehrveranstaltungen haben kann.

Name

Name des Moduls

CP

Erreichbare Credit Points des Modules. Kombiniert aus den einzelnen CP der Lehrveranstaltung

Note

Entweder 0 oder eingetragene Note

Fachsemester

Das vorhergesehene Fachsemester in welcher das Modul beelegt werden sollte.

Bestanden

Setzt sich aus der Note der LehrVeranst zusammen. Falls 4,0 oder besser = bestanden
Wie oben beschrieben besitzt das Modul eine Liste der Lehrveranstaltungen, diese sind wie folgt beschrieben:

Name

Name der Lehrveranstaltung

lvNr

Nummer der Lehrveranstaltung

Note

Double Wert für die Note

Dozent

Name des Dozenten
Das ModulEditView ist dazu da, die einzelnen Module verwalten zu können. Dazu zählt das bearbeiten, das löschen und das hinzufügen. Dieses greift auch auf die ModulListe zu, hat also alle Informationen der vorhandene Module. Dementsprechend kann dadurch von der ModulListe das aufgerufene Modul bearbeitet, löscht oder hinzufügt. (Der Unique Key ist dabei der Name des Modules)

6. Laufzeitsicht

6.1. Sequenzdiagramm Stundenplan anzeigen

Erläuterung

Das Programm wird gestartet, in der MainView wird die PlanView über set(Sub)Scene aufgerufen.
Diese fragt den Stundenplan über die StundenplanVerwaltung ab.
Gibt es zurück an unsere PlanView, welche den Stundenplan dann anzeigt.

6.2. Observer: Modulliste filtern

Erläuterung

In der ModulListView lassen sich die Module über einen Eingabestring filtern. Nach der Eingabe des Strings wird die Filtermethode der ModulListe aufgerufen. ModulListe aktualisiert sich, teilt dies dem ModulListeObserver mit und dieser kann sich anschließend die aktualisierte ModulListe abrufen und anzeigen.

6.3. User ruft auf dem Studienplan alle Module auf.

Erläuterung

Das Programm wird gestartet, in der MainView wird die PlanView über set(Sub)Scene aufgerufen. Diese fragt den Stundenplan über die StundenplanVerwaltung ab. Gibt es zurück an unsere PlanView, welche den Stundenplan dann anzeigt. Der User öffnet nun die Modul Liste (ModulListView) und bekommt alle verfügbaren Module angezeigt.

6.4. User bearbeitet Modul

Erläuterung

Der User hat die ModulListView geöffnet und möchte dort nun ein Modul bearbeiten. Er klickt auf das Modul, dann öffnet sich ein Bearbeitungsfenster(Popup). Der User macht seine Eingaben und klickt dann auf speichern. Jetzt wird der ModulController aufgerufen, welcher die bearbeitung mit der editModul() funktion durchführt. ModulListe aktualisiert sich, teilt dies dem ModulListeObserver mit und dieser kann sich anschließend die aktualisierte ModulListe abrufen und anzeigen. Die Änderung wird dann in der ModulListView angezeigt.

12. Glossar

Begriff Definition

UX – User Experience

Das zu deutsch „Nutzererlebnis“ umschreibt alle Wahrnehmungen und Emotionen eines Nutzers in Bezug auf ein Produkt oder eine Anwendung.

Usability

Eine gute Usability bedeutet, dass der User findet, was er sucht und das möglichst schnell und einfach. Im Einzelnen bedeutet das:

  • das Produkt/die Anwendung ist nützlich und bietet etwas, was der User braucht
  • der User erkennt schnell die Funktionsweisen und kann sie sich einprägen
  • der User versteht den inhaltlichen Aufbau und findet, was er sucht
  • der User braucht wenig Zeit und Energie um sein Ziel zu erreichen
  • der Gebrauch ist einfach und macht bestenfalls Spaß

UI – User Interface (Design)

Ein User Interface ist die Benutzeroberfläche einer digitalen Anwendung, z. B. einer App oder Webseite. Im UI Design wird die Gestaltung der Benutzeroberfläche definiert. Ein gutes User Interface ist im Idealfall selbsterklärend und gibt nur kleine, notwendige Hilfestellungen in menschlicher Sprache.

Komponenten

Komponenten sind Teil-Elemente des UIs. Das können z. B. einzelne Buttons oder das gesamte Main Menu oder das Sidebar Menü sein. Die Komponenten zeigen die verschiedenen Status an, die das Element hat: z. B. Default, Hover und/oder Active.

UI Element

Ein UI Element ist ein einzelnes Element der Benutzeroberfläche, z. B. ein Button, Eingabefeld oder eine Checkbox

Modul

Ein Modul ist eine UI Komponente, die den Semesterplan füllt, und Daten über die jeweilige Lehrveranstaltung Enthält, kann per Drag and Drop in den Plan integriert werden.

Drag and Drop

Die möglichkeit, eine UI-Komponente mit der Maus an eine Andere Stelle der UI zu ziehen.

Fortschrittsregelung

Jede Lehrveranstaltung hat ein Empfohlenes Fachsemester n. Um ein Modul zu belegen, muss man alle Module Bestanden haben, die für das n-3 Fachsemester bestimmt sind, also im 4. alle aus dem 1. Bestanden haben.

Stylesheet

Das einheitliche aussehen der Anwendung und ihrer Komponenten wird über ein Stylesheet definiert / festgelegt.
